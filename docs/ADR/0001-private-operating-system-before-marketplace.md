# ADR 0001: Private Operating System Before Marketplace

## Status

Accepted

## Context

International trade marketplace products fail when they try to create public liquidity before they understand real buyer, seller, quotation, and partner workflows.

TradeOS starts with operational workflows that already exist:

- inbound buyer/seller messages
- CRM follow-up
- quotation drafting
- partner/service requests
- association reporting
- approval and audit requirements

## Decision

TradeOS will be built first as a private operating system for trade teams and associations. Public marketplace features are deferred until private workflow data proves repeated matching demand.

## Consequences

Positive:

- faster path to pilot value
- stronger trust and data quality
- measurable revenue workflow before marketplace speculation
- clearer privacy boundaries

Negative:

- marketplace growth loops are delayed
- public SEO/profile features are not prioritized
- matching revenue requires later evidence threshold

## Readiness Criteria For Marketplace Work

Marketplace work may begin only when these are documented:

- repeated buyer/seller matching demand from CRM/inbox data
- opt-in data sharing model
- public/private profile boundary
- approval-gated introduction workflow
- dispute and trust process
- monetization trigger
